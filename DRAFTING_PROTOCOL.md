# Drafting Protocol

> Open this project, follow this file. It turns a blank frontier-model session into a constrained drafting workflow for your target language — one that avoids the errors documented in expert critiques of real AI Scripture drafts. The point is to make in-context constraint do the work that a fine-tuned pipeline's training data does for it — plus an output check that most pipelines never had.

## Why one-shot drafting fails (read once)

Earlier ad-hoc drafts were **one-shot, NT-only, no enforcement, no verification.** The references existed but weren't pushed into the prompt as hard rules, the proper-name table was speculative, there was no OT key-term list, no negative prompt, and nothing checked the output. Result: orthography errors, leaked related-language auxiliaries, hallucinated words, majority-language key terms. This protocol fixes the *workflow*, not just the docs.

---

## Step 1 — Load the context pack (exact order)

Load these into the drafting session, in this order (file names follow your reference pack; build them from `references/_templates/`):

1. `references/<lang>-common-errors.md` — **the negative prompt. Load this first and weight it heavily.**
1b. `references/<lang>-incontext-pack.md` — **the corpus distilled for the model:** frequency core-vocabulary + pattern banks of real verses. Load this to saturate the model in attested target-language text (examples anchor a model better than rule prose). Generated by `tools/build-incontext-pack.ts`.
2. `references/<lang>-narrative-exemplars.md` — **the positive prompt:** real attested narrative in your language, genre-matched to the passage (showing the participant-intro formula, the scene's key idioms, naming, etc.).
3. `references/<lang>-draft-annotation.md` — **how to mark uncertainty, options, and confidence** (the transparency layer that keeps the translator thinking).
3b. `references/translation-issues.md` — **the issue taxonomy + multi-option mandate:** for metaphors, measurements, unknown concepts, rhetorical structure, and discourse order, offer 2–3 grounded options, don't commit to one. (This one is a **shared, language-generic file** — it ships with the repo and has no per-language template; adapt it only if your project wants different issue policies.)
4. `references/<lang>-proper-names.md` — fixed names list + transliteration rules
5. `references/<lang>-key-terms.md` — OT key terms not in the NT
6. `references/<lang>-orthography.md` — the script / romanization system
7. `references/<lang>-grammar.md` — verb system, particles, pronouns
8. `references/<lang>-style-guide.md` — corpus vocabulary + register

> 🔒 **NEVER load the published/reference translation (`paths.check_set`) into a drafting session.** It is the CHECK set. Feeding it to the drafter contaminates the experiment. It is used only at Step 5/6 to score the finished draft.
9. **Passage-specific exemplars** — run `bun tools/find-parallels.ts -k 8 <target-language content words>` and paste the top hits. These are real corpus verses that share the scene's vocabulary — the model's best register models for *this* passage.
10. the corpus (`paths.corpus`) — the full verse-keyed corpus (a 1M-context model holds it). The model's evidence base for real target-language usage. **Curated exemplars (#2, #9) matter more than the raw dump — load both, but weight the curated.**

> The lexicon (`paths.lexicon`) is generated by `bun tools/build-lexicon.ts`. You don't paste it into the draft — it powers the verifier's corpus-membership check (Step 5). Re-run the builder if the corpus changes.

### Context loading strategy (tested, not assumed)

Examples beat rules, and curated/retrieved examples beat a raw whole-corpus dump. Measured on **one fresh verse, one model, one language** (n=1 — a directional data point consistent with the in-context-learning literature, not a benchmark; re-run the comparison on your own language before leaning on the exact numbers):

| Context | Result (verifier) |
|---------|-------------------|
| rules only | 1 error, 5 unattested words (drifted into majority/related-language forms) |
| rules + example verses (pack + retrieval) | **0 errors, 2 unattested** (reused attested forms from the loaded exemplars) |

Loading examples roughly halved the drift and killed the error. So the recommended **tiered** loading — relevance and position over raw volume:

1. **Always-on primer:** the in-context pack (#1b) + common-errors (#1) + key-terms/proper-names. Broad construction + lexical anchoring.
2. **Per-verse retrieval:** `bun tools/find-parallels.ts -k 20 <content words>` for the passage (#9) — the verses most likely to share *this* scene's vocabulary, placed up front where model attention is strongest.
3. **Full corpus as a retrieval *source*, not a raw dump.** Don't paste every verse: at scale it dilutes attention (lost-in-the-middle) and adds latency for little gain over targeted retrieval. Keep it available; retrieve from it. A few dozen scene-relevant verses teach more than thousands the model can't attend to.

## Step 2 — Generate the drafting system instruction (per language, auto)

The drafting system instruction is **no longer hardcoded.** It is **auto-generated per language** from the language profile (`profiles/<iso>.json`) and the reference pack:

```bash
bun tools/generate-system-instruction.ts
# or, for a specific profile:
LANG_PROFILE=profiles/<iso>.json bun tools/generate-system-instruction.ts
```

The tool reads the profile's `script`, `lwc`, `divine_name`, `suffixes`, `borrowings`, `key_terms_anchor`, and `linter_rules`, plus the reference pack, and emits the paste-ready system instruction: the alphabet/orthography rules, the prefer-attested-over-borrowing rules, the grammar leak corrections, the key-term anchors, the proper-name policy, the honesty rule, the discovery-before-flagging rule, the transparency/annotation mandate, and the open-cases (idiom/measurement/structure) mandate — all filled with **your** language's values instead of any one language's specifics.

Paste its output at the top of the draft prompt. Because it's generated from config, adding or changing a rule means editing the profile (not this doc and not any code): the system instruction regenerates to match. The invariant content it always emits:

- **Script/orthography** hard rules from the profile's `linter_rules` (the verifier checks the same ones).
- **Prefer attested over borrowing** from `borrowings` (majority/related-language form → the attested local form).
- **Grammar leak corrections** from the reference pack (e.g. the language's progressive/aspect marker over a leaked auxiliary; correct pronoun/suffix forms).
- **Key terms** from `key_terms_anchor` (the established renderings; the divine name).
- **Proper names:** use the fixed list; for any name not on it, propose a candidate AND mark it `⟨?⟩`.
- **Honesty rule:** if not confident a word is genuine, write it `⟨?⟩` rather than inventing a plausible-but-foreign word.
- **Discovery before flagging:** a `⟨?⟩` means "I could not find a word," not "no word exists." Before flagging any concept, run `bun tools/discover-term.ts <gloss> <synonyms...>` — it searches the LWC parallel for the concept AND its synonyms, then surfaces the ATTESTED target-language words from the aligned verses. The corpus very often has the word under a different scene. Always think of 2-3 synonyms. Confirm with `grep-corpus`, and only flag `⟨?⟩` if discovery genuinely turns up nothing usable.
- **Transparency** (see the draft-annotation spec): mark every low-confidence word `⟨?⟩`; where the text is open, give 2-3 options `{A ⟂ B}` each with its warrant (the source feature or interpretive fork that produces it); add `⟦…⟧` crux notes; append 1-2 questions (in the target language) per hard verse. Do NOT inflate certainty.
- **Open cases** (metaphor, idiom, measurement, unknown/culture concept, rhetorical structure): don't commit to one phrase — offer 2-3 warranted options and flag the issue type (see the translation-issues reference).
- **Structure:** before finalizing a narrative verse, check its discourse SHAPE against the corpus with `bun tools/find-structures.ts <type>`. Attested vocabulary in the wrong order is still wrong.

## Step 3 — The iterative loop (do NOT one-shot)

The one-shot attempt is what failed. Always iterate:

1. **Draft** the passage, with the curated exemplars (Step 1 #2, #9) in view.
1b. **Verify key forms against the corpus — don't guess.** For any non-trivial word or construction (a phrase, a verb's preposition), run `bun tools/grep-corpus.ts "<form>"` and use the *attested* construction. This is how a drafter earns a well-supported draft *before* any review, and it surfaces real choices (e.g. petition vs devotional senses of "pray") rather than guessing one.
2. **Self-audit** — instruct the model to re-read its own draft against `<lang>-common-errors.md` section by section (A alphabet → F translation choices) and list every suspected violation in its own output.
3. **Revise** based on the self-audit.
4. **Run the verifier** (Step 4) and feed the findings back; revise again.
5. **Score confidence** — `bun tools/score-draft.ts <draft.json> [--vs <other-model-draft.json>]` tags each verse HIGH/MEDIUM/LOW from objective signals (lexical attestation, `⟨?⟩` markers, cross-model agreement). This tells you and the translator **which verses to scrutinize first** — the LOW ones. It is grounded transparency, not the model's self-rating.
6. **Back-translate** — have a *different* model translate the target-language draft back into the LWC (`profile.lwc[0]`) *without looking at the source*, then diff that back-translation against the real source verse by verse. This catches the errors no spell-checker can: dropped clauses, wrong participant ("who did what to whom"), meaning drift. Any mismatch → revise. **Run it blinded** — opaque segment ids (never the verse reference), shuffled order, gloss-before-prose — because a model that recognizes the passage can back-translate from memory and silently restore the very errors this step exists to catch. (Procedure + rationale: `multi-model/backtranslate-prompt.md`.)
7. **Native-speaker / expert spot-check**, LOW-confidence verses first — **plus a random sample of unflagged verses** (2–3 per chapter minimum). The flags order the queue, but the worst error class (fluent, fully-attested, wrong) presents as *unflagged*; a spot-check that only follows the flags can never find it. Fold comments back in — and **bank every correction** (`multi-model/corrections/`) as future few-shot data.
8. Repeat 4–7 until the verifier is clean, confidence is acceptable, the back-translation matches the source, and the spot-check passes.

> Evidence this loop works (one project, one fluent reviewer — anecdotal, stated as such): a draft scored **7/10** by a fluent speaker was revised 3–4 times first; with corrections folded in it was trending toward ~8.5–9. One-shot drafts in the same project did not get those scores. The iteration *is* the method; treat the numbers as an existence proof, not a benchmark.

## Step 4 — Verify before any human reads it

```bash
bun tools/check-draft.ts <path-to-draft.json>
```

It reads the language profile and flags: script/orthography violations (the profile's `linter_rules`), vowel-length violations, leaked-auxiliary / borrowing density (`density_checks`), wrong key terms, and — against the lexicon — **any word that never appears in the corpus** (a probable hallucination, or a legitimate OT-specific term to confirm with a native speaker). Exit code is non-zero if any hard violation is found; the membership list is advisory. **No draft goes to the human reviewer until it shows no mechanical flags and you've eyeballed the unattested-word list** — the reviewer's time is for judgment calls (idioms, discourse, translation choices), not for catching mechanical leaks. ("No mechanical flags" is the entry bar, not a quality claim: every Barnwell *accuracy* category — wrong participant, dropped negation, wrong sense of an attested word — passes this verifier silently. Those are what steps 6–7 exist for.)

## Step 5 — Record the result

Save drafts under your results directory, note model + date, and record the verifier output and any expert scores. That accumulating record is the actual evidence for the "frontier vs fine-tuned pipeline" question — which is the experiment this harness exists to make runnable.

## Step 6 — Save the interlinear back-translation (consultant artifact)

> Two different "back-translations" exist; don't confuse them.
> - **The BT *check*** (Step 3 #6) is generated *live* during drafting — a different model back-translates the draft to catch dropped clauses / wrong participants / drift — and is a verification act, not a saved file. (Caveat: a frontier model that may have seen the source can produce a quietly contaminated self-BT, so it is a drift-catcher, not an independent oracle.)
> - **The interlinear BT artifact** (this step) is a *saved record* of the finished draft, for an outside consultant who does not read the target language.

Once a passage's draft is locked, generate a **strict interlinear, morphology-glossed** back-translation and save it under `multi-model/backtranslations/<book>-interlinear.md`. It is **not** natural prose — it is a linguistic record. Format, three aligned lines per verse in a monospace block:

```
TGT:   <target-language tokens, word by word>
GLOSS: <Leipzig-style morpheme gloss per token>
LIT:   '<a literal LWC rendering of the verse>'
```

Glossing standard: Leipzig-style abbreviations (1SG/3MS/3FS/3PL, PST/IMPF/PROG/IMP, DEF, NEG, Q, POSS), bound morphemes joined with `-`, clitics with `=`, sub-glosses within a word joined with `.`. Keep every `⟨?⟩` mark. This is what lets a consultant audit the draft morpheme-by-morpheme without knowing the language. (Generation is model work; the free tools do not produce glosses.)

---

## What this harness does and does not claim

It **raises the floor**: it kills the mechanical, orthography/script/related-language-leak errors that make first drafts unusable, and supplies the OT key terms and names the model otherwise invents. It does **not** certify a draft as good — the human-judgment layer (idioms, discourse, reader assistance, textual/interpretive choices) still needs a mother-tongue translator. The honest position: *one-shot frontier drafting is poor; the open question is whether a properly-resourced, iterated, verified frontier workflow beats a properly-resourced fine-tuned pipeline — and that is the experiment this harness exists to make runnable.*
