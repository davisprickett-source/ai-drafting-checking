# Language Profiles — one file to port the system to a new language

> A profile captures everything *language-specific* in one JSON, so a translation team configures the system by filling this in rather than editing code. The tools read it; the engine stays language-agnostic. The canonical shape is `profiles/_template.json` — copy it to `profiles/<iso639_3>.json` and fill it in.

## How to add a language
1. Copy `_template.json` to `<your-iso>.json` and fill it for your language.
2. Put your corpus (existing Scripture, verse-keyed JSON) at `paths.corpus` and run `bun tools/build-lexicon.ts` and `bun tools/build-incontext-pack.ts`.
3. Build the reference pack in `references/` from `references/_templates/` (orthography, grammar, common-errors, key-terms, proper-names, narrative-exemplars, style-guide) — see `WORKFLOW.md`.
4. **Write your `linter_rules[]`** (and `density_checks[]`) in the profile for your orthography and leak patterns. **This replaces the old "edit `check-draft.ts`'s `RULES[]` array" step** — there is no code edit anymore. The linter compiles its checks from the profile at runtime, so the verifier and the auto-generated drafting system instruction stay in sync from one source.
5. Point the tools at your profile with `LANG_PROFILE=profiles/<your-iso>.json` (or keep the default path), then `bun tools/generate-system-instruction.ts` to emit the drafting prompt.

## Fields

### Identity
| field | meaning |
|-------|---------|
| `language`, `iso639_3` | name + ISO 639-3 code |
| `script` | the script/orthography in prose, e.g. "Latin romanization" or "Arabic script" |
| `lwc` | ordered list of the language(s) of wider communication the team reads. `lwc[0]` is the primary — the back-translation target and the default source parallel |
| `divine_name` | how the divine name is rendered (linter & drafters use it) |

### Paths
`paths` points at every per-language data file:
| key | meaning |
|-----|---------|
| `corpus` | the verse-keyed corpus (existing Scripture) — the drafter's evidence base |
| `lexicon` | the corpus-membership oracle (generated by `build-lexicon.ts`) |
| `approved_additions` | human-confirmed forms the linter has learned (grown by `approve-form.ts`) |
| `check_set` | 🔒 the published/reference translation — **CHECK-ONLY, firewalled**, never a drafting input |
| `reference_pack` | the `references/` directory |
| `incontext_pack` | the distilled in-context pack (generated by `build-incontext-pack.ts`) |
| `references` | a `{label: path}` map of LWC source parallels the team reads, e.g. `{"<LWC> Bible": "reference-versions/<lwc-bible>.json"}`. Tools resolve the back-translation and dossier source line from here, keyed by `lwc[0]`'s label |

### Lexical / morphological config
| field | meaning |
|-------|---------|
| `suffixes` | pronominal suffixes that attach without a hyphen (drives the linter's de-hyphenation + hyphen check) |
| `borrowings` | a map of majority-language / related-language forms → the attested local form to prefer (linter suggestions + drafting prompt) |
| `key_terms_anchor` | the established renderings of core biblical terms (e.g. God, the LORD, Temple, priest, sacrifice, pray) |
| `stopwords` | high-frequency function words to exclude from content-word retrieval (`find-parallels`, `discover-term`) so matches key on meaningful vocabulary |
| `vowels` | the language's vowel inventory (drives the vowel-length / max-identical-vowels check) |
| `core_word_pattern` | a regex defining what counts as a single well-formed word (used to tokenize and to flag stray characters); e.g. `^[\\p{L}'ʼ]+$` |
| `book_aliases` | a map of book-name aliases → canonical book name, so verse keys from different sources normalize (e.g. `{"<short name>": "<canonical name>"}`) |

### Linter rules (replaces the old code edit)
`linter_rules[]` — an array of regex checks the verifier runs against a draft. Each entry:
| key | meaning |
|-----|---------|
| `id` | short stable id for the rule (shown in verifier output) |
| `pattern` | the regex that matches the **bad** form |
| `flags` | regex flags, e.g. `"gi"` |
| `severity` | `"error"` (hard; non-zero exit) or `"warn"` (advisory) |
| `lexExempt` | if `true`, a match is suppressed when the matched word is attested in the lexicon/approved-additions (so a legitimately-attested form isn't false-flagged) |
| `message` | what is wrong and what to use instead (shown to the user) |

This is where the per-language orthography and leak checks live (script substitutions, marked-character rules, hyphen-suffix rules, etc.). Adding or tightening a check is a config edit here — the verifier and `generate-system-instruction.ts` both read it.

**You do not need to write regex yourself.** If regex isn't your thing, paste this into your AI session: *"Interview me about my language's orthography and the majority/related-language interference we actually see in drafts, then write the `linter_rules[]` and `density_checks[]` entries for my profile. After writing each rule, test it: show me 3 strings it should catch and 3 it should not, and run the linter on a sample to confirm."* Then verify each rule fires (and doesn't over-fire) on a real text before trusting it — a wrong rule that never fires is invisible, and one that over-fires trains the team to ignore the linter.

### Density checks
`density_checks[]` — flags a discouraged form that is *overused relative to* a preferred form (the classic leaked-auxiliary tell). Each entry:
| key | meaning |
|-----|---------|
| `id` | short stable id |
| `form` | regex for the discouraged form |
| `flags` | regex flags for `form` |
| `vs` | regex for the preferred form it competes with |
| `vsFlags` | regex flags for `vs` |
| `severity` | `"error"` or `"warn"` |
| `message` | what the imbalance means |

The check fires when the discouraged form's count is high relative to the preferred form's — catching drift a single-match rule would miss.

### In-context patterns
`incontext_patterns[]` — named constructions `build-incontext-pack.ts` mines from the corpus to build the pattern banks (real-verse examples of each construction). Each entry:
| key | meaning |
|-----|---------|
| `name` | short label for the construction (e.g. "participant introduction") |
| `note` | one line on what the construction is / when to use it |
| `pattern` | regex matching the construction in corpus verses |
| `flags` | regex flags |
| `n` | how many example verses to include in the pack |

### Structure patterns
`structure_patterns[]` — discourse-shape patterns `find-structures.ts` retrieves by, so a drafter can check a verse's *shape* (not just its vocabulary) against how the corpus does that scene type. Each entry:
| key | meaning |
|-----|---------|
| `id` | short stable id (the `<type>` argument to `find-structures.ts`) |
| `desc` | what the discourse type is (e.g. narrative-opening, participant-intro, dialogue-frame, temporal-when) |
| `pattern` | regex matching that shape in corpus verses |
| `flags` | regex flags |

## The two invariants (enforced, not optional)
- **Firewall:** `paths.check_set` is CHECK-ONLY — never loaded into a drafting session, never copied into `references/`.
- **Growing linter:** only human-confirmed forms enter `approved_additions` (via `approve-form.ts`). Never AI output, never the check set.

## The portability promise
Everything that differs between languages is **data in the profile + the reference pack** — including the linter rules. The method (tiers, council, dossier, consultant-check, flag levels), the renderers, and **all** tools are language-agnostic. Adding a language is a configuration task, not a re-implementation.
